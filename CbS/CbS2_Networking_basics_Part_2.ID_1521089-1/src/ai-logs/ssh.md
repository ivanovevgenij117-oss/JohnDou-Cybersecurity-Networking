# Анализ механизмов безопасности протокола SSH и TCP-Handshake

## 1. Разбор TCP-флагов и согласования Sequence Numbers
В процессе анализа дампа `ssh.pcap` перед инициализацией прикладного уровня SSH зафиксировано классическое трехстороннее рукопожатие (TCP Three-Way Handshake) со следующей структурой флагов и порядковых номеров:

1. **Пакет SYN (Клиент -> Сервер)**:
   - *Флаги*: Установлен строго бит `SYN` (Synchronize, значение `0x002`).
   - *Sequence Number*: Клиент генерирует начальное случайное число $X$ (Initial Sequence Number / ISN), отображаемое в Wireshark как относительный ноль `0`.
2. **Пакет SYN-ACK (Сервер -> Клиент)**:
   - *Флаги*: Одновременно взведены биты `SYN` и `ACK` (`0x012`), подтверждая готовность к двусторонней синхронизации.
   - *Sequence Number*: Сервер генерирует собственный случайный номер $Y$.
   - *Acknowledgment Number*: Равен $X + 1$ (относительная `1`), подтверждая получение SYN-запроса от клиента.
3. **Пакет ACK (Клиент -> Сервер)**:
   - *Флаги*: Установлен строго бит `ACK` (Acknowledgment, значение `0x010`).
   - *Sequence Number*: Переходит в состояние $X + 1$.
   - *Acknowledgment Number*: Равен $Y + 1$, завершая фазу установления надежной сессии.

*Соответствие теории:* Логика полностью соответствует спецификации RFC 793. Использование относительных номеров в Wireshark упрощает верификацию шагов приращения байт.

## 2. Описание этапов установления SSH-сессии
После успешного TCP-рукопожатия протокол SSHv2 инициализирует защищенный канал в четыре шага:
1. **Exchange of Version Strings**: Устройства обмениваются текстовыми строками с версиями ПО для подтверждения совместимости.
2. **Algorithm Negotiation (KEXINIT)**: Клиент и сервер обмениваются списками поддерживаемых криптографических примитивов (симметричное шифрование, хэш-функции, обмен ключами).
3. **Key Exchange (KEX)**: На основе алгоритма Диффи-Хеллмана и сгенерированных на Cisco RSA-ключей (1024 бит) устройства вычисляют общий секретный ключ сессии (Session Key) без его прямой передачи по сети. На этом этапе проверяется цифровая подпись хоста для предотвращения атак типа **Man-in-the-Middle (MITM)**.
4. **User Authentication**: Весь последующий трафик инкапсулируется в состояние `Encrypted Packet`. Передача логина, хэшированного пароля и команд администрирования происходит внутри изолированного криптографического туннеля.

## 3. Сравнительный ИБ-анализ трафика Telnet и SSH
На основе пакетного анализа сделаны следующие выводы о различиях протоколов удаленного управления:

| Параметр сравнения | Протокол Telnet (порт TCP 23) | Протокол SSH (порт TCP 22) |
| :--- | :--- | :--- |
| **Формат передачи данных** | Открытый текст (Plain Text) | Бинарный зашифрованный поток (Ciphertext) |
| **Видимость в Wireshark** | Полный текст команд, логинов и паролей доступен через *Follow TCP Stream*. | Пакеты отображаются как `Encrypted Application Data`. Содержимое скрыто. |
| **Целостность данных** | Отсутствует. Уязвим к модификации пакетов на лету. | Обеспечивается механизмами HMAC (контроль целостности каждого кадра). |

### Критические риски использования Telnet:
1. **Уязвимость к сниффингу (Credential Theft)**: Хакер в том же широковещательном домене может легко перехватить пароли администратора.
2. **Отсутствие аутентификации хоста**: Telnet не защищает от подмены сервера (MITM), позволяя злоумышленнику перехватить сессию управления инфраструктурой.
