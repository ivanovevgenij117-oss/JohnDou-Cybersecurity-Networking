# ИБ-Отчет: Взаимодействие с AI-ассистентом и архитектурный аудит VLAN Hopping

## 1. Анализ конфигурации AI и обнаружение критических различий
При генерации базового конфига ИИ-ассистент выдал классический набор команд, однако в процессе ручной отладки на оборудовании Cisco 3745 (модуль EtherSwitch NM-16ESW) были обнаружены два важнейших несоответствия:

- **Различие 1 (Именование интерфейсов):** AI предложил использовать интерфейсы `fa0/0` и `fa0/1`. В реальности на Cisco 3745 нулевой слот занят портами маршрутизации L3. Модуль коммутации `NM-16ESW` физически размещается в `Slot 1`. Использовались интерфейсы `FastEthernet 1/1` и `FastEthernet 1/2`.
- **Различие 2 (Поведение утилиты nonegotiate):** AI выдал команду `switchport nonegotiate` для ручного отключения протокола DTP на портах доступа. Однако при вводе роутер выдал ошибку `% Invalid input detected`. 

---

## 2. Техническое обоснование ошибки AI и специфика оборудования Cisco
В ходе инженерного анализа причин ошибки `Invalid input` было установлено следующее:

1. **Особенность модуля NM-16ESW:** В старых коммутационных чипах Cisco (таких как NM-16ESW) ввод базовой команды `switchport mode access` **автоматически и намертво отключает протокол DTP** на аппаратном уровне. Порт физически перестает генерировать динамические кадры согласования. Поэтому команда `nonegotiate` здесь заблокирована операционной системой как избыточная.
2. **Специфика современных платформ Cisco (Catalyst / Nexus):** В современном сетевом оборудовании Cisco поведение изменилось. Даже при жесткой фиксации `switchport mode access` порт продолжает в фоновом режиме слать DTP-пакеты. Из-за этого современные коммутаторы остаются уязвимыми к атаке *Switch Spoofing*. В современном продакшене ИБ-инженеры **обязаны** явно прописывать `switchport nonegotiate` для закрытия этой уязвимости периметра.

## 3. Итоговая конфигурация защиты от VLAN Hopping
Поскольку DTP на клиентских портах защищен «из коробки» самим фактом ввода `mode access`, главным вектором атаки оставалось двойное тегирование (*Double Tagging*) на Trunk-магистрали. 

Для защиты периметра был изменен дефолтный Native VLAN 1 на изолированный скрытый номер на обоих устройствах.

### Конфигурация интерфейса Trunk на SW1 и SW2:
```text
interface FastEthernet 1/0
 switchport trunk encapsulation dot1q
 switchport mode trunk
 ! Защита от Double Tagging: переносим нетегированный трафик на пустой VLAN 999
 switchport trunk native vlan 999
 no shutdown
end
write memory
```

### Обоснование выбора команд:
Если хакер попытается отправить с компьютера «отравленный» кадр с двумя тегами (внешний VLAN 1 и внутренний VLAN 20 жертвы), коммутатор `SW1` примет его. Но поскольку Native VLAN изменен на 999, коммутатор не станет стирать внешний тег. Пакет улетит в изолированный VLAN 999 и физически не сможет перепрыгнуть на порт жертвы в VLAN 20.

## 4. Автоматизация аудита сети через GNS3 REST API (Python)
Ниже представлен скрипт для автоматической проверки безопасности конфигураций коммутаторов в проекте:

```python
import requests

GNS3_SERVER = "http://127.0.0"
PROJECT_ID = "vlan_assignment_uuid"

def run_audit():
    nodes = requests.get(f"{GNS3_SERVER}/projects/{PROJECT_ID}/nodes").json()
    for node in nodes:
        if "SW" in node["name"]:
            config = requests.get(f"{GNS3_SERVER}/projects/{PROJECT_ID}/nodes/{node['node_id']}/files/vios_config.txt").text
            
            # Проверка безопасности Native VLAN
            if "native vlan 1" in config or "native vlan" not in config:
                print(f"[{node['name']}] УЯЗВИМОСТЬ: Native VLAN 1 активен! Возможен VLAN Hopping.")
            else:
                print(f"[{node['name']}] Безопасность Trunk: ОК (Native VLAN изменен на изолированный)")

if __name__ == '__main__':
    run_audit()
```
